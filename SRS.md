## 1. סקירה כללית

### 1.1 מטרת המערכת
מערכת ניהול אירוע רפואי המיועדת למנהלי אירוע רפואי (Case Managers) לניהול מסע המטופל מרגע האבחון ועד ההחלמה, כולל ניהול ביטוחים, תביעות, מסמכים, ואסטרטגיה פיננסית-רפואית.

### 1.2 קהל יעד
| תפקיד | תיאור |
|--------|--------|
| מנהל ראשי (Admin) | גישה מלאה, ניהול משתמשים |
| מנהל אירוע רפואי (Manager) | ניהול מטופלים ותהליכים |
| מטופל (Patient) | צפייה במסע הרפואי האישי |

### 1.3 טכנולוגיות
- **Backend:** FastAPI (Python), SQLAlchemy, SQLite
- **Frontend:** React + Vite + Tailwind CSS
- **Deployment:** Railway (Production), GitHub Actions (CI/CD)
- **Auth:** JWT + 2FA (TOTP / Email)

## 2. מצב נוכחי — מה קיים במערכת

### 2.1 מודולים פעילים

#### 🔐 אימות ואבטחה
- הרשמה / התחברות עם JWT
- אימות דו-שלבי (2FA) — TOTP ואימייל
- שחזור סיסמה
- ניהול פרופיל משתמש
- הגנת נתיבים (ProtectedRoute)

#### 👥 ניהול מטופלים
- יצירה / עריכה / מחיקה של מטופלים
- שיוך קופת חולים ורמת ביטוח
- פרטי אבחנה ותיאור מקרה
- חיפוש וסינון

#### 🗺️ מסע מטופל (Patient Journey)
- ציר זמן אחוד עם 5 שלבים קבועים + צמתים מותאמים אישית
- שלבים קבועים: גילוי ואבחון, תכנון הטיפול, שלב הטיפולים, החלמה ושיקום, מעקב
- עריכה inline: תאריך, הערות, מיקום מחדש
- סטטוס לכל צומת: עתידי / פעיל / הושלם

#### 🏥 ביטוחים (Insurance)
- מקורות ביטוח: סל הבריאות, קופ"ח, הר הביטוח, פרטי, ביטוח לאומי
- ניהול כיסויים לפי קטגוריה (ניתוח, אשפוז, שיקום, אבחון, מחלות קשות וכו')
- פרטי כיסוי: סכום, אחוז, השתתפות עצמית, תקרה שנתית, כיסוי בחו"ל

#### 📋 תביעות (Claims)
- יצירה ומעקב תביעות מול מקורות ביטוח
- סטטוסים: ממתין / הוגש / אושר / חלקי / נדחה
- שדות: סכום מבוקש, סכום מאושר, תאריך הגשה, דדליין, עדיפות
- קישור למקור ביטוח

#### 💡 אסטרטגיה (Strategy / Entitlements)
- ניהול זכאויות: קיימות, פוטנציאליות, צפויות
- תיאור, סכום, סטטוס אישור

#### 📄 מסמכים (Documents)
- העלאת מסמכים לפי מטופל (PDF, תמונות, Word, Excel — עד 20MB)
- קטגוריות: רפואי, ביטוחי, משפטי, פיננסי, אחר
- הורדה ומחיקה
- אחסון: `/data/uploads/{patient_id}/` ב-Railway

#### 👨‍⚕️ מאגר רופאים
- ~370 רשומות ממקורות מגוונים (CKAN, Google Sheets, אתרים)
- נרמול אוטומטי (שם, מומחיות, תת-מומחיות)
- סינון, חיפוש, ייצוא Excel
- ייבוא PDF / Excel

#### ⭐ ציוני רספונסיביות
- דירוג חברות ביטוח וקופות חולים
- מדדים: מהירות תגובה, רמת בירוקרטיה, ציון כולל

#### 💬 משובים (Feedback)
- טופס משוב ציבורי (Landing Page)
- תיבת דואר נכנס למנהל

#### 🔧 ניהול (Admin)
- ניהול משתמשים (admin בלבד)
- הצגת / מחיקת משתמשים

## 3. דרישות פונקציונליות — פיתוח עתידי

### 3.1 מנוע זרימת עבודה (Flow Engine) ⭐ עדיפות גבוהה

#### תיאור
מנוע State Machine המאפשר הגדרה, הפעלה, ומעקב אחר תהליכים מובנים (Workflows) לניהול תהליכי טיפול רפואי ותביעות ביטוח.

#### מושגי יסוד
| מושג | תיאור |
|------|--------|
| **Workflow Template** | תבנית זרימה ניתנת לשימוש חוזר (למשל: "הגשת תביעה לקופ״ח") |
| **Workflow Instance** | הרצה ספציפית של תבנית עבור מטופל מסוים |
| **Step / State** | שלב בזרימה (למשל: "איסוף מסמכים", "הגשה", "המתנה") |
| **Transition** | מעבר בין שלבים (עם תנאים / טריגרים) |
| **Action** | פעולה אוטומטית או ידנית הנדרשת בשלב |
| **Assignment** | שיוך שלב למשתמש / תפקיד |

#### תבניות זרימה מוגדרות מראש
1. **הגשת תביעת ביטוח** — איסוף מסמכים → הגשה → מעקב → תוצאה
2. **קבלת חוות דעת שנייה** — חיפוש רופא → קביעת פגישה → קבלת חוות דעת → שילוב בתיק
3. **ערר על דחייה** — ניסוח ערר → הגשה → שמיעה → תוצאה
4. **תהליך אשפוז** — רישום מוקדם → כניסה → שחרור → מעקב
5. **הגשה לביטוח לאומי** — בדיקת זכאות → איסוף טפסים → הגשה → ועדה רפואית

#### דרישות מפורטות
- **FR-WF-01:** יצירת תבניות workflow מותאמות אישית
- **FR-WF-02:** הרצת workflow עבור מטופל ספציפי
- **FR-WF-03:** מעקב סטטוס שלב בזמן אמת
- **FR-WF-04:** הגדרת תנאי מעבר (אוטומטי / ידני)
- **FR-WF-05:** שיוך משימות לאנשים
- **FR-WF-06:** תזכורות ודדליינים לכל שלב
- **FR-WF-07:** היסטוריית פעולות (Audit Log)
- **FR-WF-08:** חיבור workflow לתביעה / מסמך / צומת במסע המטופל
- **FR-WF-09:** דוח סיכום workflow עם ציר זמן

#### מודל נתונים מוצע
```
WorkflowTemplate: id, name, description, steps (JSON), created_by
WorkflowInstance: id, template_id, patient_id, current_step, status, started_at, completed_at
WorkflowStep: id, instance_id, step_name, step_order, status, assignee_id, due_date, completed_at, notes
WorkflowAction: id, step_id, action_type, triggered_by, timestamp, data (JSON)
```

## 4. דרישות לא פונקציונליות

| קטגוריה | דרישה |
|---------|--------|
| **אבטחה** | JWT עם תפוגה, 2FA חובה, HTTPS ב-production |
| **ביצועים** | תגובת API < 500ms לשאילתות רגילות |
| **זמינות** | 99.5% uptime דרך Railway |
| **נגישות** | ממשק RTL מלא בעברית |
| **אחסון** | Volume מתמיד ב-Railway לDB ולמסמכים |
| **גיבוי** | גיבוי DB אוטומטי (לא מיושם עדיין) |
| **הרחבות** | ארכיטקטורה מודולרית — הוספת routes / modules ללא שינוי ליבה |
