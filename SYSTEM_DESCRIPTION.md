# Orly Medical — מנהל אירוע רפואי
## תיאור המערכת | גרסה 2.0 | אפריל 2026

---

## 1. סקירה כללית

Orly Medical היא מערכת ניהול אירוע רפואי (Medical Case Management) המיועדת למנהלי אירוע רפואי המלווים מטופלים לאורך תהליכים רפואיים מורכבים. המערכת מנהלת את מסע המטופל מרגע הקליטה ועד ההחלמה, ומספקת ניהול מלא של ביטוחים, תביעות, מסמכים, זרימות עבודה מובנות, ואסטרטגיה פיננסית-רפואית.

### 1.1 קהל יעד

| תפקיד | תיאור | הרשאות |
|--------|--------|---------|
| מנהל ראשי (Admin) | גישה מלאה לכל המערכת | ניהול משתמשים, מחיקה, הגדרות |
| מנהל אירוע (Manager) | ניהול מטופלים ותהליכים | CRUD מלא על מטופלים, זרימות, תביעות |
| מטופל (Patient) | צפייה במסע האישי | קריאה בלבד — מתוכנן לגרסה עתידית |

### 1.2 ארכיטקטורה טכנית

| שכבה | טכנולוגיה | תפקיד |
|------|-----------|--------|
| Backend | FastAPI (Python) + SQLAlchemy | API, לוגיקת עסקים, מסד נתונים |
| Database | SQLite + Railway Volume | אחסון מתמיד ב-/data |
| Frontend | React + Vite + Tailwind CSS | ממשק RTL בעברית |
| Auth | JWT + 2FA (TOTP / Email) | אימות ואבטחה |
| Deploy | Railway + GitHub Actions | CI/CD אוטומטי לכל push ל-main |
| Production | https://app-production-5817.up.railway.app | כתובת Production |

## 2. מודולים קיימים

### 2.1 אימות ואבטחה

- הרשמה והתחברות עם JWT (תוקף 30 יום)
- אימות דו-שלבי (2FA) — TOTP (Google Authenticator) ואימייל
- שחזור סיסמה עם קישור בתוקף 1 שעה
- ניהול פרופיל משתמש (שם, אימייל, סיסמה, מפתח 2FA)
- הגנת נתיבים — ProtectedRoute עם הפניה אוטומטית לדף כניסה

### 2.2 ניהול מטופלים

- יצירה, עריכה ומחיקה של מטופלים
- שיוך קופת חולים ורמת ביטוח (בסיסי / משולם / פרימיום / זהב)
- פרטי אבחנה, מצב רפואי, ותגיות מצב רפואי (condition tags)
- חיפוש וסינון דינמי
- 5 טאבים לכל מטופל: פרטים, ביטוחים, תביעות, אסטרטגיה, מסמכים

### 2.3 מסע מטופל (Patient Journey)

ציר זמן אחוד המציג את כל שלבי הטיפול הרפואי של המטופל:

- **6 שלבים קבועים:** קליטת מטופל ← גילוי ואבחון ← תכנון הטיפול ← שלב הטיפולים ← החלמה ושיקום ← מעקב
- **צמתים מותאמים אישית:** ניתן להוסיף אירועים בין השלבים (ניתוח, ייעוץ, תביעה...)
- **עריכה inline:** תאריך, הערות, מיקום מחדש בין השלבים
- **סטטוס לכל צומת:** עתידי / פעיל / הושלם
- **סנכרון עם מנוע הזרימה:** השלמת שלב בזרימה → עדכון אוטומטי של הצומת המתאים

### 2.4 ביטוחים (Insurance)

- **מקורות ביטוח:** סל הבריאות, קופת חולים, הר הביטוח, ביטוח פרטי, ביטוח לאומי
- **קטגוריות כיסוי:** ניתוח, אשפוז, שיקום, אבחון, השתלה, מחלות קשות, טכנולוגיה מתקדמת, חוות דעת שנייה
- **פרטי כיסוי:** סכום מכוסה, אחוז כיסוי, השתתפות עצמית, תקרה שנתית, כיסוי בחו"ל
- **יועץ כיסוי אוטומטי:** חישוב כיסוי לכל שלב בזרימת העבודה

### 2.5 תביעות (Claims)

- יצירה ומעקב תביעות מול כל מקורות הביטוח
- **סטטוסים:** טיוטה / ממתין / הוגש / אושר / חלקי / נדחה
- **יצירה אוטומטית** של תביעות טיוטה ע"י מנוע הזרימה לשלבים פיננסיים
- שדות: סכום מבוקש, סכום מאושר, תאריך הגשה, דדליין, עדיפות
- קישור ישיר לשלב בזרימת עבודה

### 2.6 מסמכים (Documents)

- העלאת מסמכים לפי מטופל (PDF, תמונות, Word, Excel — עד 20MB)
- קטגוריות: רפואי, ביטוחי, משפטי, פיננסי, אחר
- הורדה ומחיקה מאובטחת
- אחסון בנפח מתמיד: `/data/uploads/{patient_id}/` על Railway

### 2.7 מאגר רופאים

- כ-370 רשומות ממקורות מגוונים: CKAN ממשלתי, Google Sheets, אתרים רפואיים
- נרמול אוטומטי: שם, מומחיות, תת-מומחיות, טלפון, מיקום
- סינון לפי מומחיות, קבלת קופ"ח, מתן חוות דעת מומחה
- ייבוא PDF / Excel, ייצוא Excel עם עיצוב RTL ועברית

### 2.8 ציוני רספונסיביות

- דירוג חברות ביטוח וקופות חולים לפי איכות שירות
- מדדים: מהירות תגובה (1-10), רמת בירוקרטיה (1-10), ציון כולל
- משמש לדירוג מקורות ביטוח מומלצים בזרימות העבודה

### 2.9 אסטרטגיה ייעוצית

- ניהול זכאויות: קיימות, פוטנציאליות, צפויות
- המלצות תביעות לפי שלב טיפולי ומצב המטופל
- מטריצת כיסויים — סיכום ויזואלי של כל הביטוחים
- תובנות לימוד מובנות למנהלי אירוע

## 3. מנוע זרימת עבודה (Flow Engine)

מנוע הזרימה הוא הלב האוטומציה של המערכת — State Machine המאפשר הגדרה, הפעלה, ומעקב אחר תהליכים מובנים לכל מטופל.

### 3.1 מושגי יסוד

| מושג | תיאור |
|------|--------|
| Workflow Template | תבנית זרימה ניתנת לשימוש חוזר (למשל: "הגשת תביעה ביטוחית") |
| Workflow Instance | הרצה ספציפית של תבנית עבור מטופל מסוים |
| Step / שלב | שלב בזרימה עם סטטוס: ממתין / פעיל / הושלם / דולג |
| Task / משימה | פריט ב-checklist שיש להשלים בתוך שלב |
| Audit Log | רשומת כל פעולה: מי, מתי, מה — בלתי ניתן למחיקה |
| Coverage Advisor | מחשב כיסוי ביטוחי אוטומטית לכל שלב שמופעל |
| Force Advance | מעבר שלב מאולץ על ידי מנהל — עוקף משימות ונרשם בלוג |

### 3.2 לוגיקת State Machine

כל Instance נמצא תמיד במצב מוגדר. מעברים חוקיים:

| פעולה | מצב לפני | מצב אחרי | תנאי |
|--------|-----------|-----------|------|
| advance_step() | active | הבא active / completed | כל המשימות הושלמו |
| advance_step(force=True) | active | הבא active / completed | ללא בדיקת משימות + לוג |
| skip_step() | active | skipped | שלב אופציונלי בלבד |
| pause_instance() | active | paused | תמיד |
| resume_instance() | paused | active | תמיד |
| cancel_instance() | active/paused | cancelled | עם סיבה |

### 3.3 תבניות מובנות

| תבנית | קטגוריה | שלבים | מטרה |
|--------|---------|-------|------|
| מסע המטופל | journey | 6 | תהליך מלא מקליטה עד מעקב — נוצר אוטומטית |
| הגשת תביעה ביטוחית | claim | 5 | תביעה לביטוח פרטי |
| ערר על דחיית תביעה | appeal | 7 | הגשת ערר לאחר דחייה |
| קבלת חוות דעת שנייה | treatment | 6 | מציאת מומחה וחוות דעת |
| הגשה לביטוח לאומי | claim | 7 | נכות / ניידות / שירותים מיוחדים |
| תהליך אשפוז מתוכנן | hospitalization | 9 | מהכנה טרום-אשפוז עד מעקב |
| פתיחת תיק אונקולוגי | treatment | 7 | ניהול ראשוני עם אבחנת סרטן |
| מעקב טיפול כימותרפי | treatment | 5 | ניהול מחזורי כימותרפיה |
| הכנה להשתלת איבר | treatment | 7 | ניהול תהליך ההשתלה המלא |
| שיקום לאחר שבץ מוחי | treatment | 6 | שיקום וזכויות לאחר שבץ |

### 3.4 Coverage Advisor — יועץ כיסוי ביטוחי

בכל פעם שמופעל שלב חדש, המערכת מחשבת אוטומטית:

- **התאמת קטגוריות:** התאמת קטגוריות כיסוי של השלב לביטוחים הפעילים של המטופל
- **חישוב פיננסי:** סכום מכוסה, פער (gap), אחוז כיסוי לכל מקור
- **דירוג:** מיון מקורות לפי ציוני רספונסיביות — מי כדאי לתבוע קודם
- **המלצה בעברית:** הצעה מה לעשות ומאיזה מקור
- **תביעת טיוטה:** יצירה אוטומטית לשלבים פיננסיים עם כיסוי מזוהה

## 4. תהליך קליטת מטופל (Patient Intake)

שלב קליטת המטופל הוא השלב הראשון במסע המטופל — נפתח אוטומטית עם יצירת מטופל חדש. שלב זה חייב להיות מושלם לפני מעבר לשלב הבא.

### 4.1 ארבע משימות החובה

| # | משימה | תיאור |
|---|-------|--------|
| 1 | רישום פרטי המטופל ומידע ראשוני | שם מלא, ת"ז, פרטי קשר, מצב רפואי ראשוני |
| 2 | איסוף פוליסות ביטוח | זיהוי כל הביטוחים הפעילים: קופ"ח, פרטי, הר הביטוח, ב"ל |
| 3 | קביעת פגישת היכרות | תיאום פגישה ראשונה עם המטופל והמשפחה |
| 4 | חתימת הסכם ייפוי כוח | חתימה על ייפוי כוח והסכם ניהול האירוע |

### 4.2 זרימת השלב

| שלב | פעולה | תוצאה |
|-----|--------|--------|
| יצירת מטופל חדש | המנהל יוצר מטופל | Instance מסע מטופל נפתח + שלב קליטה פעיל |
| השלמת משימות | סימון checkboxes ✓ | Checklist מתעדכן בזמן אמת |
| כל המשימות הושלמו | לחיצה על "השלם שלב" | מעבר אוטומטי לגילוי ואבחון |
| משימות פתוחות | לחיצה על "אלץ מעבר" | מעבר עם תיעוד בלוג + אזהרה ויזואלית |

### 4.3 מסע המטופל המלא

| step_order | שלב | תיאור | משימות |
|------------|-----|--------|--------|
| 5 | קליטת מטופל | אונבורדינג ראשוני | 4 משימות חובה |
| 10 | גילוי ואבחון | ממצאים ואבחנה רשמית | — |
| 20 | תכנון הטיפול | בחירת מסלול טיפולי + חוות דעת שנייה | — |
| 30 | שלב הטיפולים | אשפוז, ניתוח, כימותרפיה | — |
| 40 | החלמה ושיקום | פיזיותרפיה, חזרה לתפקוד | — |
| 50 | מעקב | ביקורי מעקב תקופתיים | — |

## 5. ממשקי API עיקריים

### 5.1 Auth

- `POST /api/auth/register` — הרשמה
- `POST /api/auth/login` — התחברות + בדיקת 2FA
- `POST /api/auth/verify-2fa` — אימות קוד 2FA (TOTP / אימייל)
- `POST /api/auth/forgot-password` — שליחת קישור שחזור סיסמה
- `POST /api/auth/reset-password` — איפוס סיסמה עם טוקן

### 5.2 Patients

- `GET /api/patients` — רשימת מטופלים עם פילטרים
- `POST /api/patients` — יצירת מטופל חדש (+ journey workflow)
- `GET/PUT/DELETE /api/patients/{id}` — קריאה / עדכון / מחיקה
- `GET/POST/PUT/DELETE /api/patients/{id}/nodes` — ניהול צמתים

### 5.3 Workflows

- `GET /api/workflows/templates` — רשימת תבניות
- `POST /api/workflows/instances` — יצירת זרימה חדשה
- `GET /api/workflows/instances?patient_id={id}` — זרימות לפי מטופל
- `POST /api/workflows/instances/{id}/steps/{step_id}/advance` — השלמת שלב
- `POST /api/workflows/instances/{id}/steps/{step_id}/tasks/{task_id}/toggle` — סימון משימה
- `POST /api/workflows/instances/{id}/steps/{step_id}/skip` — דילוג שלב אופציונלי
- `POST /api/workflows/instances/{id}/pause` — השהיית זרימה
- `POST /api/workflows/instances/{id}/resume` — חידוש זרימה
- `POST /api/workflows/instances/{id}/cancel` — ביטול זרימה
- `GET /api/workflows/suggest?patient_id={id}` — המלצות זרימות

### 5.4 Claims / Insurance / Documents

- `GET/POST /api/claims` — תביעות
- `PUT /api/claims/{id}` — עדכון סטטוס ופרטים
- `GET/POST /api/insurance` — מקורות ביטוח וכיסויים
- `POST /api/documents/upload` — העלאת מסמך
- `GET /api/documents/{patient_id}` — רשימת מסמכים
- `GET /api/documents/download/{filename}` — הורדת מסמך
